# Synthesis & recommendation

## 1. Top-line recommendation

**Pivot, then ship a narrower MVP.** DuoData targets a real bottleneck — *metric literacy* as the unmeasured upstream of Performance Clarity ≥75% — and F053 (Cross-Functional Connections) is genuinely uncopiable IP. But the PRD as written is two products glued together: a capability-infrastructure bet (F053 + F054 + live DataHub binding) and a consumer-style gamified habit shell (hearts, leagues, R7/R30 targets) imported from Duolingo without the audience-fit work. The shell is mismatched to a captive, finite, senior-skewed audience of a few hundred Amanoters, and the success metrics are framed in product-app cohort semantics on a closed population — exactly the act-date-vs-cohort confusion the tool claims to teach. Cut to the moat, fix four blocking issues before any code ships, and run a 4-month bench that proves the F053 pre/post-test gap moves before committing to the full path map.

## 2. Convergent findings (2+ of 4 agents agreed)

- **Success metrics are wrong for a closed, captive audience.** Critic B2, Hypotheses H-A1/H-G1, Evaluation §1 (stronger replacements), and Strategic §1 all converge: R1/R7/R30/WAU framed on cohort semantics misapplies product-app retention to an L&D tool with no acquisition funnel. Replace with path completion within onboarding cycle, time-to-first-correct-dashboard-read, manager-attested readiness, and the F053 pre/post-test gap. Performance Clarity, not R7, is the headline KPI.
- **Cross-Functional Connections (F053) is the moat — everything else is commodity.** Critic ranks F053 inconsistencies as B1 (highest), Strategic §2 and §3 name it as the only non-rentable component, Hypotheses H-C1 marks the pre/post-test as the load-bearing claim, Evaluation §2 calls the pre/post gap "the single most important learning metric." If F053 lands, DuoData is justified; if not, buy 360Learning/Sana.
- **Content rot is the modal failure mode and the drift checker is hand-waved.** Critic M3 (string-diff false positives), Hypotheses H-C4 (DataHub schema stability unvalidated), Evaluation §3 (definition-drift anti-metric), Strategic §6 (de-risk move #2: live DataHub binding, IronSource shutdown as forcing function). F100 AC-F100-06 needs structured `metric_references` foreign keys, not free-text string compare.
- **Senior staff / gamification mismatch is a real adoption risk, not a copy fix.** Critic M1, Hypotheses H-A3 and H-G2, Strategic §4 (cut leagues/hearts/daily-quests from MVP). "Expert path that skips intros" does not mitigate hearts blocking a P3 UA Lead's lesson on a numeric-tolerance miss. Default-off gamification for IC4+/M2+, or replace hearts with a non-blocking mastery score.
- **Definition drift and equation inconsistency in F053/00-overview break the canonical-symbol promise.** Critic B1 (`Profit = Installs × (UV − eCPI)` vs `LTV − eCPI` vs `ARPDAU × DAU − Cost`), Hypotheses H-C2 (act-date vs cohort is the most-mistaught concept), Strategic §3 (the tacit IP is the metric tree as Amanotes uses it). The headline equation must be written once, parameterized as UVn, and referenced from a single ADR.
- **Authoring sequencing is internally inconsistent.** Critic M7 (F100 in parallel with F050–F053 but `blocks` says otherwise), Strategic §4 (DnI authors via Markdown PR for MVP, defer authoring UI), Hypotheses H-C3 (DRIs are not pedagogy experts — rework risk). Either ship F100 first, or author content in YAML/Markdown and cut F100 to phase 2.

## 3. Divergent findings — and my position

- **Build the gamification stack now, or strip it?** Critic M1 says strip / role-gate; Strategic §4 says cut hearts/leagues/quests entirely from MVP; Hypotheses H-G1 calls it LOW confidence; Evaluation §3 lists multiple gamification anti-metrics. The Author claims gamification is "justified, not decorative." **Position: strip it.** Keep XP and streak (low cost, removable), cut hearts (F021 blocking behavior), leagues (F022), and league-driven daily quests from MVP. Re-introduce after the pre/post-test bench proves learning is happening — gamification on top of a working learning loop is additive; gamification masking no learning is the worst outcome.
- **Is MEP partner expansion the killer follow-on, or a distraction?** Strategic §5 says MEP partner enablement is the biggest follow-on bet; Hypotheses H-A4 marks it LOW confidence and unvalidated; Critic n2 calls out the framing overselling. **Position: name MEP as the C3/C4'26 expansion thesis but do not let it shape MVP scope.** SSO scope, content tenancy, and partner-specific paths are all phase-2+ — Critic m2 (OQ-F001-01 `@amanotesjsc.com`) is the only MVP-blocking partner concern.
- **Pre/post-test target: ≥80% from ≤40% baseline — realistic or aggressive?** Hypotheses H-C1 calls 40→80 aggressive (MEDIUM confidence); Evaluation §5 sets the *sunset* threshold at <15pp gap and *pause* at <30pp; Strategic §6 says 70% on the bench would be enough to keep building. **Position: re-baseline the target as ≥30pp absolute lift, with ≥70% absolute floor on F053 items, rather than the 40→80 framing.** Run the pre-test now on n=20 Amanoters (per H-C1 cheapest test) to set a real baseline before MVP commits.
- **Replace R30 entirely, or condition it on `card_mastered`?** Evaluation §1 proposes conditioning R30 on mastered cards; Critic B2 wants R30/R7 dropped from the success-metric framing. **Position: drop R7/R30 as success metrics, keep them as engagement-curve diagnostics only, and elevate `card_mastered` density and pre/post-test gap to the success tier.** This is the cleanest fix to the act-date-vs-cohort dogfooding problem.

## 4. PRD changes required before approval (Must-fix)

Replace with: (a) F053 pre/post-test gap ≥30pp absolute lift with ≥70% post floor; (b) new-hire path-completion-within-onboarding-cycle ≥60% (existing KPI, kept); (c) time-to-first-correct-dashboard-read p50 ≤14 days (from Evaluation §2); (d) `card_mastered` density ≥7 across days 0–30 cohort. R1/R7/R30 demoted to "engagement curve, not retention" with the act-date-vs-cohort caveat made explicit. [Critic B2, Evaluation §1]
2. **Fix the canonical equation in F053 U1 and 00-overview.** Pick `Profit = Installs × (UVn − eCPI)` with `n` parameterized (UV7/UV30/UV365). 4. Does Amanotes IT/Legal have a DPIA process for an employee-performance tool storing 365 days of per-card scores in BigQuery? If yes, who is the named owner; if no, we are inventing one.
5. What is the actual addressable headcount in PI/BH/DR/GDUC squads, broken out by role × grade? The 30% WAU target and the H-A1 / H-A3 sample sizes both depend on this; the PRD never states it.

## 6. Pre-launch validation plan — 4 cheapest tests

All five can run in C1'26, before any production code on F050+, at well under one engineer-cycle. Each ties back to a hypotheses.md row.

1. **F053 pre-test bench, n≈20, paper prototype.** Run the F053 U1+U2 item set on 20 Amanoters across PO/UA/MO/DnI today, with no intervention. [H-C1, Strategic §6] *Decision rule:* if baseline >50%, the 40→80 framing is fiction and we re-baseline the target. If <30%, the gap is huge and the thesis is even stronger. Either way, this is the single most decision-changing test in the plan.
2. **DataHub glossary diff.** Pull DataHub's glossary today, diff against `/prd/glossary.md`, count canonical-string mismatches and OOV terms. [H-C4] *Decision rule:* if drift count >20%, F100 drift-checker must use structured references (Must-fix #5) and the live-binding architecture is non-negotiable. If <5%, snapshot-with-monthly-sync is a viable phase-1 fallback.
3. **DRI authoring rework test.** Ask 3 DRIs (one each from Product/UA/Mon) to author 5 cards in YAML against the F050 quality bar. Run the metric-name-allowlist linter and a peer review. [H-C3] *Decision rule:* if rework rate >40% on first submission, DnI is the bottleneck — either accept slower content velocity or build a heavier authoring UI (F100) ahead of content.
4. **Senior-staff perception interviews, n≈5.** 30-min sessions with 3 M2 PMs + 2 P3 GDs on a clickable mock that includes both hearts-blocking and a non-blocking mastery variant. Score perceived-childishness and willingness-to-do-daily. [H-A3, H-G2] *Decision rule:* if ≥3/5 score the hearts variant as "I would close the tab," Must-fix #4 is validated and the gamification strip is permanent, not a phase-1 holdover.
5. **Manager tutoring time diary, n=3 M2s.** 1-week pre-MVP time diary on metric-tutoring hours. [H-O2] *Decision rule:* if baseline is <1 hr/week, the "save manager time" JTBD is overclaimed and we cut it from the strategy narrative. If ≥3 hr/week, the 6-week post-launch repeat is the strongest possible 6-mo defense.

## 7. Recommended decision

- **Ship MVP: yes, conditional on the C1'26 bench (Plan §6 test 1) returning ≥20pp gap potential.** If the bench shows <20pp possible lift, kill the build and rent Sana/360Learning for Product/UA/Mon 101 — the moat doesn't justify the engineering cost.
- **Scope cut: yes, aggressively.** Ship F050 (Product 101) + F053 U1+U2 only + minimum-viable player (F004 text/MCQ, no scenario branching, no hearts) + F054 (static SVG metric tree, P0) + DataHub canonical-string linter + Metabase tracking dashboard (F101 reduced to adoption + learning dashboards only). Cut: F051 (UA 101), F052 (Mon 101), F053 U3–U5, F021 hearts, F022 leagues, F023 quest mechanic, F100 authoring UI (replace with Markdown-PR authoring by DnI), mobile, Vietnamese, MEP partner SSO scope.

**1-cycle (4-month, C2'26) action plan:**

- **Weeks 1–2 (C1 close):** Run all 5 validation tests in §6. Lock or kill on test 1's result.
- **Weeks 3–4:** Rewrite 00-overview success metrics, write ADR-006/007/008, fix F053 U1 equation, strip gamification from F004/F021. PRD re-review.
- **Weeks 5–10:** Build minimum-viable player + F050 Product 101 + F053 U1+U2 + DataHub linter + static F054 tree. DnI authors content via Markdown PR.
- **Weeks 11–13:** Closed pilot with 1 PI squad's next 5 new hires (H-A2). Run F053 post-test on pilot cohort.
- **Weeks 14–16:** Decision review against Evaluation §5 thresholds. If post-test gap ≥30pp and path completion ≥50% in pilot squad, greenlight F051 + F052 + F053 U3–U5 for C3'26. If gap is between 15–30pp, content-redo pause per Evaluation §5. If <15pp, sunset and rent.

The point of this cycle is not to ship a Duolingo clone. It is to prove that *Amanotes-specific metric literacy* is teachable, and that DnI can land Performance Clarity as a measurable KPI. Everything else — gamification, mobile, MEP — is a downstream bet that only pays out if this single result lands.
